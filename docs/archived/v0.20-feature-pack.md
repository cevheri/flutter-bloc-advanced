## 2. Token Refresh + Secure Session

**What:** Automatic JWT token refresh that queues concurrent requests during refresh.

**How it works:**
- `TokenRefreshInterceptor` (QueuedInterceptor) catches 401 responses
- Queues all pending requests while refreshing the token
- On success: updates storage, retries all queued requests
- On failure: triggers session expiry callback → redirects to login
- `FlutterSecureStorageAdapter` replaces plaintext JWT storage
- `SecurityUtils.isTokenExpired()` now actively checks JWT `exp` claim

**Files:**
- `lib/infrastructure/http/interceptors/token_refresh_interceptor.dart`
- `lib/infrastructure/storage/secure_storage.dart`
- `lib/core/security/security_utils.dart` (activated)

## 4. Analytics + Crash Reporting

**What:** Pluggable analytics interface — swap providers by changing one DI registration.

**How it works:**
- `IAnalyticsService` (abstract) defines: `logScreenView`, `logEvent`, `logUserAction`, `logError`, `setUserId`, `setUserProperty`

- `AnalyticsBlocObserver` automatically logs every BLoC transition
- `AnalyticsRouteObserver` automatically logs every screen navigation

**To switch to Firebase:** Implement `IAnalyticsService` with Firebase calls, swap in DI.

**Files:**

## 5. Repository Cache + Offline Data

**What:** HTTP response caching with 4 policies — previously loaded data survives offline.

**Cache policies:**
| Policy | Behavior |
|--------|----------|
| `networkFirst` | Try network, fall back to cache on error (default) |
| `cacheFirst` | Try cache, fetch from network if expired |
| `networkOnly` | Never cache |
| `cacheOnly` | Never fetch from network |

**How it works:**
- `CacheInterceptor` sits in the Dio chain, caches successful GET responses
- `SharedPrefsCacheStorage` stores entries with metadata + TTL
- On network failure: serves stale cache with `x-cache: stale` header
- Default TTL: 5 minutes (configurable per request)

**Files:**
- `lib/infrastructure/cache/cache_storage.dart` (interface + CachePolicy enum)
- `lib/infrastructure/cache/shared_prefs_cache_storage.dart` (implementation)
- `lib/infrastructure/http/interceptors/cache_interceptor.dart`

## 7. State Time-Travel Debugging

**What:** Rewind, replay, and export BLoC state transitions.

**Capabilities:**
- **Timeline** — vertical chronological list of all state changes, filterable by BLoC
- **State Diff** — tap any snapshot to see before/after state comparison
- **Rewind** — restore a BLoC to any previous state (UI updates instantly)
- **Replay** — auto-play transitions at 800ms intervals
- **Export** — copy entire timeline as JSON (attach to bug reports)

**How it works:**
- `TimeTravelBlocObserver` records every transition with a reference to the actual state object
- `TimeTravelStore` maintains per-BLoC ring buffers (100 snapshots each)
- Rewind calls `bloc.emit(previousState)` — no BLoC code changes needed
- `kDebugMode` guard — disabled in release builds

**Files:** `lib/app/dev_console/time_travel/`

## 8. Smart Retry + Circuit Breaker

**What:** Self-healing network layer that automatically retries transient failures and prevents cascade failures.

**Retry logic:**
- Retries: timeout, connection error, 502, 503, 504
- Does NOT retry: 400, 401, 403, 404, 422
- Max 3 attempts with exponential backoff + jitter (200ms → 400ms → 800ms)
- Respects `Retry-After` header

**Circuit breaker (per endpoint):**
```
[CLOSED] ──5 failures──→ [OPEN] ──30s cooldown──→ [HALF-OPEN]
   ↑                        │                          │
   │                   instant fail               1 probe request
   └──────success────────────────────────success───┘
```

- Each API endpoint has its own circuit breaker
- **Open** state: requests fail instantly (~0ms instead of 30s timeout)
- **Half-Open** state: allows one probe request to test recovery
- `CircuitBreakerOpenException` tells the UI exactly which endpoint is down

**Files:**
- `lib/infrastructure/http/circuit_breaker.dart` (pure Dart state machine)
- `lib/infrastructure/http/interceptors/resilience_interceptor.dart`

## 10. Dynamic Forms Engine

**What:** A JSON schema renders into a fully functional, validated, themed Flutter form.

**16 supported field types:**
`text` · `email` · `password` · `number` · `phone` · `textarea` · `dropdown` · `multiSelect` · `date` · `datetime` · `toggle` · `checkbox` · `radio` · `slider` · `sectionHeader` · `divider`

**Example schema** (`assets/mock/GET_dynamic_forms_pathParams.json`):
```json
{
  "id": "create_lead",
  "title": "New Lead",
  "fields": [
    {"type": "text", "key": "name", "label": "Full Name", "required": true, "validators": ["minLength:2"]},
    {"type": "email", "key": "email", "label": "Email", "required": true},
    {"type": "dropdown", "key": "source", "label": "Source", "options": ["Web", "Referral"]},
    {"type": "toggle", "key": "priority", "label": "High Priority", "default": false}
  ],
  "submitAction": {"method": "POST", "endpoint": "/leads"}
}
```

**How it works:**
- `FormSchemaModel.fromJsonString()` parses JSON → entity
- `DynamicFormRenderer` maps each field type to the project's design system widgets
- Validators: `required`, `minLength:N`, `maxLength:N`, `email`, `numeric`
- Submit sends data to the schema's `submitAction.endpoint`
- Route: `/dynamic-forms/:formId`

**Files:** `lib/features/dynamic_forms/`

## Interceptor Chain (Complete)

The Dio HTTP client now has 8 interceptors in this order:

```
Request →
  1. ConnectivityInterceptor    — instant offline rejection
  2. AuthInterceptor            — injects JWT Bearer token
  3. TokenRefreshInterceptor    — handles 401, refreshes token
  4. ResilienceInterceptor      — retry + circuit breaker
  5. MockInterceptor            — (dev/test only) serves mock JSON
  6. CacheInterceptor           — stores/serves cached responses
  7. DevConsoleInterceptor      — records for debug panel
  8. LoggingInterceptor         — structured console logging
→ Response
```
